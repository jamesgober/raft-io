# raft-io v0.2.0 — The Deterministic Core

**The first release with consensus logic.** v0.2.0 turns the v0.1 scaffold into a
working protocol core: a `RaftNode` you drive with one `step(event) -> Vec<Action>`
function, leader election with full term and vote safety, the leader heartbeat,
and single-node commit — all with no wall clock and no I/O. Election Safety is
checked by a `proptest` suite over randomised schedules, and the `step` hot path
has a `criterion` baseline. No durable storage, no multi-node replication yet —
those are v0.3 and v0.4.

## What is raft-io?

A from-scratch implementation of the Raft consensus algorithm, built as a clean,
embeddable library rather than a framework. The protocol core is a deterministic
state machine: you feed it events (logical ticks, inbound messages, client
proposals) and it returns actions (send these messages, apply this committed
command). Time, networking, and storage are injected through trait seams, which
is what makes the core provable and reproducible from a seed. It is the
consensus layer above `wal-db` and the coordination substrate for Hive DB
clustering.

## What's new in 0.2.0

### `RaftNode` and the `step` function — the whole protocol through one door

Everything funnels through one method:

```rust
use raft_io::{Action, Event, RaftConfig, RaftNode};

let mut node = RaftNode::new(RaftConfig::single(1));

// Logical time is the caller's: feed ticks at whatever interval you choose.
while !node.is_leader() {
    let _ = node.step(Event::Tick).expect("tick never fails in memory");
}

// A single-node leader commits its own proposals at once (quorum of one).
let actions = node.step(Event::Propose(b"set x = 1".to_vec())).unwrap();
assert!(actions.iter().any(|a| matches!(a, Action::Apply { .. })));
assert_eq!(node.commit_index(), 1);
```

`step` returns `Result<Vec<Action>>`. The node decides *what* must happen — send
a message, apply a command — and the caller carries it out, in the order
returned. Nothing inside reads a clock, opens a socket, or touches a disk.

### Leader election with term and vote safety

The election rules are implemented in full: a node increments its term and votes
for itself when it stands, grants at most one vote per term, applies the
up-to-date-log election restriction before granting, and steps down the moment it
sees a higher term. Election timeouts are randomised per node — but from a
**seeded, deterministic** generator (SplitMix64), so a cluster jitters like real
Raft while every run stays reproducible.

```rust
use raft_io::{Action, Event, Message, RaftConfig, RaftNode, RequestVote, Role};

let mut node = RaftNode::new(RaftConfig::new(1, [2, 3]));

// A vote request from a candidate with an up-to-date log is granted once…
let granted = |n: &mut RaftNode, candidate| {
    n.step(Event::Message(Message::RequestVote(RequestVote {
        term: 5, candidate, last_log_index: 0, last_log_term: 0,
    })))
    .unwrap()
    .iter()
    .any(|a| matches!(a,
        Action::Send { message: Message::RequestVoteReply(r), .. } if r.vote_granted))
};
assert!(granted(&mut node, 2));
assert!(!granted(&mut node, 3)); // …and refused to a second candidate in term 5
assert_eq!(node.role(), Role::Follower);
```

### `RaftConfig` — a one-call common case, a builder for tuning

```rust
use raft_io::RaftConfig;

// Tier 1: defaults (10..=20 tick election timeout, 3 tick heartbeat).
let cfg = RaftConfig::new(1, [2, 3]);

// Tier 2: tune timing for a deployment that ticks every millisecond.
let tuned = RaftConfig::new(1, [2, 3])
    .with_election_timeout(150, 300)
    .with_heartbeat_interval(30)
    .with_seed(0xABCD);
# assert_eq!(tuned.election_timeout(), (150, 300));
```

Out-of-order or zero arguments are normalised rather than rejected, so a
misconfiguration cannot wedge a node.

### The `RaftLog` and `RaftTransport` seams

The two trait seams the protocol is built around are defined now, each with an
in-memory implementation:

- **`RaftLog`** — log entries plus the persistent `HardState` (term and vote),
  with a `sync` durability barrier. `MemoryLog` is the reference, non-durable
  implementation; a `wal-db`-backed log plugs in here in v0.4. A node adopts its
  log's persisted `HardState` on construction, so recovery is already wired.
- **`RaftTransport`** — message delivery. `MemoryTransport` records outgoing
  messages so a test harness routes them by hand.

```rust
use raft_io::{HardState, MemoryLog, RaftConfig, RaftNode, RaftLog};

// A node resumes in the term and vote its log last persisted.
let mut log = MemoryLog::new();
log.set_hard_state(HardState { term: 7, voted_for: Some(3) }).unwrap();
let node = RaftNode::with_log(RaftConfig::single(1), log);
assert_eq!(node.term(), 7);
```

### Error type on `error-forge`

The crate's `Error` is `#[non_exhaustive]` and implements
`error_forge::ForgeError`, so consumers get the portfolio's stable `kind` /
`caption` / severity metadata. A proposal to a non-leader returns
`Error::NotLeader { leader }` carrying the known leader for redirection — a
routine, retryable condition, distinct from a fatal `Error::Storage`.

### Property-based testing — Election Safety

`tests/safety.rs` builds an in-memory cluster and drives it with randomised
schedules of ticks and deliveries, asserting Raft's defining election invariant:

- **`election_safety_three_nodes`** — no two leaders in one term, over 256
  randomised schedules on a 3-node cluster.
- **`election_safety_five_nodes`** — the same on a 5-node cluster.
- **`cluster_converges_on_a_single_leader`** — the liveness counterpart: a fair
  schedule converges on exactly one leader.

Because the core is deterministic, any counterexample proptest finds is exactly
replayable.

### Benchmarks — a baseline for the `step` hot path

`benches/raft_bench.rs` measures the three shapes of `step` that dominate a
running cluster. Indicative numbers (Windows x86_64, release):

| Benchmark | Time | Notes |
|---|---|---|
| `step_follower_tick` | ~7 ns | steady-state tick, allocation-free |
| `step_handle_request_vote` | ~48 ns | election hot path |
| `step_single_node_propose` | ~128 ns | append + commit + apply |

### `API.md` — the full public surface

`docs/API.md` documents every public item — value types, `RaftConfig`,
`RaftNode` and `step`, `Event` / `Action`, the messages, both trait seams and
their in-memory implementations, and the error type — each with a description,
its parameters, and runnable examples, organised by the three API tiers.

## Breaking changes

This is the first release with a real API surface; there is nothing released
before it to break. One change relative to the v0.1 scaffold's *intended* shape:
the `std` feature and the `persistence` / `framing` flags were removed, deferred
to the phases that introduce code using them (v0.4 / v0.5). The core takes a hard
dependency on `error-forge`, so it is std-based.

## Verification

Run on Windows x86_64, Rust stable; the same commands pass on Linux (WSL2
Ubuntu) and via the CI matrix:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
cargo deny check
cargo audit
cargo bench
```

All green. Test counts at this tag:

- 44 unit tests
- 3 integration / property tests (`tests/safety.rs`)
- 36 doctests

`loom` is not yet exercised: the v0.2 core is a single-threaded, owned state
machine with no lock-free or shared-state path, so there is nothing for a model
checker to explore. It arrives when a concurrent driver does.
